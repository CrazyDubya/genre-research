# Deliverable Specifications

Clear deliverable specifications eliminate ambiguity about what "done" looks like. This guide covers defining, documenting, and validating deliverables.

## The Deliverable Framework

### What Constitutes a Deliverable

A proper deliverable is:
- **Tangible** - Something that can be reviewed
- **Complete** - Contains all required components
- **Formatted** - Meets technical specifications
- **Named** - Follows naming conventions
- **Documented** - Accompanied by any required notes

### Types of Writing Deliverables

#### Manuscript Deliverables
- Full manuscript
- Partial manuscript (milestone)
- Chapter drafts
- Scene drafts
- Revision drafts

#### Planning Deliverables
- Story outline
- Character profiles
- World-building documents
- Plot timeline
- Research notes

#### Supporting Deliverables
- Synopsis (various lengths)
- Query letter
- Back cover copy
- Chapter summaries
- Scene breakdown

## Specification Components

### Format Specifications

```markdown
## Format Requirements

### File Format
- Primary: .docx (Microsoft Word compatible)
- Alternative: .md (Markdown) with specified processor
- Never: .pdf for editable content

### Document Setup
- Font: Times New Roman, 12pt
- Line spacing: Double
- Margins: 1 inch all sides
- Paragraph: Indent first line 0.5 inch, no extra space between paragraphs

### Page Setup
- Page breaks: Between chapters only
- Headers: Author name / Title / Page number (after first page)
- Chapter headings: Centered, bold, Chapter + number

### Naming Convention
- Pattern: [ProjectCode]_[Type]_[Version]_[Date]
- Example: ROMSUSP_MS_v1.2_20240115.docx
- Milestone example: ROMSUSP_M3_v1.0_20240115.docx
```

### Content Specifications

```markdown
## Content Requirements

### Word Count
- Target: [Specific number]
- Acceptable range: [Min] - [Max]
- Count method: Microsoft Word word count (excluding front matter)

### Structure
- Chapters: [Number range]
- Average chapter length: [Range] words
- Scenes per chapter: [Range typical]
- POV: [Specification]
- Tense: [Specification]

### Required Elements
- [Element 1]: [Specification]
- [Element 2]: [Specification]
- [Additional elements as needed]

### Excluded Elements
- [Restriction 1]: [Reason/context]
- [Restriction 2]: [Reason/context]
```

### Quality Specifications

```markdown
## Quality Requirements

### Technical Quality
- Grammar: Publication-ready (minimal errors)
- Spelling: US English, consistent
- Punctuation: Chicago Manual of Style
- Formatting: Consistent throughout

### Prose Quality
- Voice: [Specification]
- Pacing: [Specification]
- Readability: [Grade level or description]
- Engagement: [Measurable criteria]

### Story Quality
- Plot coherence: No holes, logical progression
- Character consistency: Motivations and behaviors aligned
- World consistency: Rules followed, details accurate
- Emotional impact: [Specification]
```

## Revision Expectations

### Revision Round Structure

```markdown
## Revision Framework

### Round 1: Developmental
Focus: Big-picture story issues
- Structure and pacing
- Character arcs
- Plot holes
- Thematic consistency

Expected changes: Significant rewriting possible
Timeline: [Duration]

### Round 2: Line Edit
Focus: Prose-level improvements
- Sentence structure
- Word choice
- Voice consistency
- Scene-level pacing

Expected changes: Moderate revision
Timeline: [Duration]

### Round 3: Copyedit Response
Focus: Technical corrections
- Grammar fixes
- Spelling corrections
- Consistency issues
- Style guide compliance

Expected changes: Surgical corrections
Timeline: [Duration]

### Round 4: Final Proofread
Focus: Final polish
- Typo catching
- Formatting verification
- Last consistency check

Expected changes: Minimal
Timeline: [Duration]
```

### Revision Response Requirements

For each revision round:

```markdown
## Revision Response Expectations

### Documentation Required
- Track changes: ON for all modifications
- Comments: Response to each editorial comment
- Summary: Brief revision summary document

### Response Types
- Accept: Change made as suggested
- Accept with modification: Change made with variation (explain)
- Stet: Original retained (must justify)
- Query: Clarification needed before action

### Response Format
- Address every editorial note
- No deletions of editorial comments
- Clear status for each item
- Summary count: Accepted/Modified/Stet/Query
```

## Acceptance Criteria

### Milestone Acceptance

```markdown
## Milestone Acceptance Criteria

### Automatic Acceptance (All must be true)
- [ ] Word count within 10% of milestone target
- [ ] All required scenes/chapters included
- [ ] Format specifications met
- [ ] Named correctly
- [ ] Previous feedback addressed

### Quality Threshold (Most must be met)
- [ ] Prose quality consistent with samples
- [ ] Story logic holds
- [ ] Character voice maintained
- [ ] Pacing appropriate for milestone
- [ ] No glaring errors

### Review Triggers (Any of these triggers detailed review)
- Word count significantly off target
- Quality noticeably different from previous
- Previous feedback not addressed
- New concerning patterns
```

### Final Acceptance

```markdown
## Final Acceptance Criteria

### Non-Negotiable Requirements
- [ ] Complete manuscript delivered
- [ ] Word count in acceptable range
- [ ] All structural requirements met
- [ ] Format specifications satisfied
- [ ] All revision rounds completed

### Quality Requirements
- [ ] Prose meets established standards
- [ ] Story coherent start to finish
- [ ] Characters consistent and developed
- [ ] Genre requirements satisfied
- [ ] Technical quality publication-ready

### Market Requirements
- [ ] Target audience served
- [ ] Genre expectations met
- [ ] Comparable title alignment
- [ ] Commercial viability confirmed
```

## Rejection Criteria

### What Triggers Rejection

```markdown
## Rejection Criteria

### Immediate Rejection
- Plagiarism detected
- Fundamentally different from agreed concept
- Quality far below acceptable threshold
- Critical requirements ignored

### Revision Required
- Quality below threshold but fixable
- Missing required elements
- Structural problems
- Consistency issues

### Conditional Acceptance
- Minor issues that can be fixed in copyedit
- Small deviations from specification
- Stylistic concerns within acceptable range
```

### Rejection Communication

When rejecting:
1. Be specific about rejection reasons
2. Reference relevant specifications
3. Provide path to acceptance if possible
4. Maintain professional, constructive tone
5. Document decision and rationale

## Documentation Requirements

### Deliverable Tracking

Maintain record of:
- What was delivered
- When it was delivered
- Format and specifications met
- Acceptance status
- Any conditions or notes

### Version Control

```markdown
## Version Tracking

### Version Numbering
- Major versions: v1.0, v2.0 (significant changes)
- Minor versions: v1.1, v1.2 (moderate changes)
- Patches: v1.1.1, v1.1.2 (small fixes)

### Change Documentation
- What changed
- Why it changed
- Who requested
- Date of change

### Archive Requirements
- Retain all major versions
- Retain milestone submissions
- Retain final accepted version
- Retain key correspondence
```
